# Что умеет проект и как работает поиск цены КПП

Краткое описание возможностей и полный путь от первого сообщения клиента до ответа с ценой на контрактную коробку передач.

---

## Что умеет проект

Проект — это **B2B SaaS для автоматизации продаж контрактных КПП** (автомат, механика, вариатор). Операторы общаются с клиентами в Telegram / WhatsApp / MAX; система помогает подбирать коробку по машине клиента и подсказывает цены и готовые ответы.

**Основные возможности:**

1. **Приём первого сообщения клиента**  
   Клиент может написать:
   - **VIN** (17 символов) или **номер кузова (FRAME)** в тексте;
   - или отправить **фото**: шильдик КПП (код коробки) или свидетельство о регистрации ТС (оттуда извлекается VIN/FRAME).

2. **Определение коробки по машине**  
   По VIN/FRAME система:
   - обращается к **Подзамену** и **PartsAPI** (параллельно);
   - получает **OEM-код коробки** (например `2500A230`) и/или **модель коробки** (например `W5MBB`);
   - собирает контекст: марка, модель, год, тип КПП (МКПП/АКПП/вариатор), привод и т.д.

3. **Поиск цены на контрактную КПП**  
   По OEM (и при необходимости по модели коробки) система ищет в интернете объявления о продаже б/у и контрактных КПП, собирает цены и выдаёт клиенту **диапазон цен** (мин–макс, средняя) и варианты ответа оператору.

4. **Ответы оператору**  
   Система предлагает оператору готовые формулировки (шаблоны): диапазон цен, варианты «эконом / средний / премиум» по пробегу, способы оплаты и т.д. Оператор может отправить ответ одним кликом или отредактировать.

5. **Если цен не найдено**  
   Включена **эскалация**: оператору создаётся задача с готовыми поисковыми запросами для ручного поиска. Либо (если эскалация отключена) система может дать **оценку цены от ИИ** с пометкой низкой уверенности.

## Как всё работает от первого сообщения

### Шаг 1. Сообщение клиента

- Клиент пишет в чат (Telegram/WhatsApp/MAX) или отправляет фото.
- Входящее сообщение обрабатывает **inbound-message-handler**: из текста извлекается VIN или FRAME, либо по фото вызывается классификация (шильдик КПП / свидетельство о регистрации).

### Шаг 2. Что может быть в сообщении

| Ввод клиента | Действие системы |
|--------------|------------------|
| **Текст с VIN (17 символов)** | Проверка контрольной суммы; при необходимости автоисправление. Создаётся **задача на поиск по VIN** (vehicle lookup). |
| **Текст с номером кузова (FRAME)** | Создаётся задача на поиск по FRAME (тот же пайплайн, но idType = FRAME). |
| **Фото шильдика КПП** | Распознаётся код коробки (например W5MBB) → сразу ставится в очередь **поиск цены по OEM** (без шага «по VIN»). |
| **Фото свидетельства о регистрации** | Из изображения извлекается VIN/FRAME → создаётся задача vehicle lookup, как при текстовом VIN. |
| **Неполный VIN (16 символов)** | Клиенту отправляется подсказка: «Проверьте, обычно VIN из 17 символов». |
| **Текст без VIN/FRAME и без подходящего фото** | Запускается обычный AI-диалог (подсказки, ответы), без специализированного поиска КПП. |

### Шаг 3. Поиск по VIN/FRAME (vehicle lookup)

1. Создаётся **кейс** в БД и задача в очередь **vehicle_lookup_queue**.
2. Воркер параллельно вызывает:
   - **Подзамену** (Python-сервис) — по VIN/FRAME возвращает данные по машине и **коробке**: OEM-код, модель коробки, заводской код и т.д.;
   - **PartsAPI** (partsapi.ru) — декодирует VIN и даёт марку, модель, год, тип КПП, привод и т.д.
3. Из ответов собирается **vehicleContext**: марка, модель, год, тип КПП (МКПП/АКПП/вариатор), модель коробки (подсказка), привод и др. При необходимости тип КПП/привод доопределяется через GPT по сырым полям PartsAPI.
4. Результат кешируется (**vehicle_lookup_cache**), кейс помечается как выполнен.
5. **Дальше возможны три ветки:**
   - **Высокая уверенность и есть OEM** → в очередь **price_lookup_queue** ставится поиск цены по **OEM + подсказка модели + vehicleContext**.
   - **Только модель коробки (без OEM), например VW** → в очередь ставится поиск цены **без OEM**, с fallback по марке/модели/типу КПП.
   - **Нет OEM** → запускается **fallback-поиск цены** по марке, модели и типу КПП.

Если и Подзамена, и PartsAPI не нашли данные (например неверный VIN), кейс завершается с ошибкой (NOT_FOUND / PARSE_FAILED).

### Шаг 4. Поиск цены на КПП (price lookup)

Здесь как раз и строится **дерево поиска цены**. Входные данные: OEM (или null при fallback), подсказка модели коробки (oemModelHint), vehicleContext.

1. **Ключ кеша**  
   Строится ключ вида `oem::марка::модель` (в нижнем регистре).

2. **Проверка глобального кеша цен**  
   В БД ищется актуальный **price_snapshot** по этому ключу (с учётом срока годности: 7 дней для найденных цен, 24 часа для «не найдено», 2 часа для оценки ИИ).  
   Если кеш есть → клиенту сразу формируется подсказка с уже сохранённым диапазоном цен (или шаблоном «эконом/средний/премиум»), **поиск в интернете не выполняется**.

3. **Определение модели коробки по OEM**  
   Если пришла валидная подсказка модели (например W5MBB) — она используется как есть.  
   Иначе вызывается **identifyTransmissionByOem**: сначала проверяется кеш **transmission_identity_cache**, при промахе — GPT-4.1 с веб-поиском, чтобы получить «рыночное» название коробки (как в объявлениях: F4A42, W5MBB, JF011E и т.д.). Результат: modelName, производитель, происхождение (Япония/Европа и т.д.), уверенность.

4. **Поиск цен в интернете** — см. дерево ниже.

5. **После получения результата поиска**  
   - Если **найдено достаточно объявлений** → результат сохраняется в **price_snapshots** (глобальный кеш), формируются подсказки оператору (диапазон цен, при достаточном количестве — эконом/средний/премиум по пробегу).
   - Если **ничего не найдено** (not_found):  
     - при включённой эскалации → создаётся **задача оператору** с готовыми поисковыми запросами;  
     - иначе при включённой оценке ИИ → вызывается **estimatePriceFromAI** (GPT с веб-поиском), результат сохраняется с коротким TTL и низкой уверенностью;  
     - иначе клиенту отправляется сообщение «цен не найдено».

## Дерево поиска цены коробки передач

Ниже — как именно ищется цена после шага «определение модели коробки» (вызов **searchUsedTransmissionPrice**).

```
searchUsedTransmissionPrice(oem, modelName, origin, vehicleContext, tenantId)
│
├─ 1. Получить курсы валют (fetchLiveFxRates) — для международного fallback.
│
├─ 2. Сформировать поисковые запросы (buildPrimaryQuery, buildFallbackQuery):
│      например: "контрактная МКПП W5MBB 2500A230 Mitsubishi Lancer б/у из Японии",
│                "контрактная МКПП W5MBB 2500A230 Mitsubishi Lancer цена купить".
│
├─ 3. ЭТАП 1 — Yandex + Playwright (основной источник)
│   │
│   ├─ 3.1. buildYandexQueries: несколько запросов (МКПП/АКПП/вариатор + OEM + марка/модель + "контрактная").
│   │
│   ├─ 3.2. searchYandex(query) по каждому запросу → объединение результатов, сортировка по приоритету доменов
│   │        (baza.drom.ru, farpost.ru, avito.ru, japancar.ru и т.д.).
│   │
│   ├─ 3.3. Взять до 8 URL с наивысшим приоритетом → открыть до 5 страниц через Playwright (или fetch).
│   │
│   ├─ 3.4. На каждой странице: парсинг HTML (cheerio), извлечение объявлений и цен (parseListingsFromHtml),
│   │        фильтр по ключевым словам (коробка, контрактная, б/у — включить; гидроблок, на запчасти — исключить),
│   │        дедуп по URL, удаление выбросов по IQR (removeOutliers).
│   │
│   └─ 3.5. Условие успеха: ≥ 3 объявлений ИЛИ ≥ 2 разных доменов.
│            ДА → вернуть результат с source: "yandex", min/max/avg по оставшимся ценам → сохранить снимок, подсказки → КОНЕЦ.
│            НЕТ → перейти к этапу 2.
│
├─ 4. ЭТАП 2 — GPT с веб-поиском (fallback)
│   │
│   ├─ 4.1. Проверка флага GPT_WEB_SEARCH_ENABLED. Выключен → вернуть not_found → эскалация/оценка ИИ/сообщение «не найдено».
│   │
│   ├─ 4.2. runSearch(primaryQuery): GPT-4.1 с инструментом web_search ищет в русском интернете цены на контрактную КПП,
│   │        в ответе ожидается JSON-массив: { price (RUB), source, title }. Парсинг (parseListingsFromResponse), проверка цен (validatePrices).
│   │
│   ├─ 4.3. Фильтр: исключить объявления с признаками «новая», «восстановленная», «дефект», «на запчасти» и т.д.
│   │
│   ├─ 4.4. Если после фильтра < 2 объявлений → runSearch(fallbackQuery) (второй вариант запроса), снова фильтр.
│   │
│   ├─ 4.5. removeOutliers по ценам (IQR), пересчёт min/max/avg. Если после выбросов < 2 объявлений → not_found.
│   │
│   └─ 4.6. Иначе → source: "openai_web_search", min/max/avg, список объявлений → сохранить снимок, подсказки → КОНЕЦ.
│
└─ 5. Если на этапе 2 ничего не нашли — попытка международного поиска (Yahoo Auctions, eBay, JDM и т.д.)
       с конвертацией в рубли по курсам. Если и там пусто → not_found → эскалация / оценка ИИ / «не найдено».
```

**Кратко по этапам:**

- **Этап 1 (Yandex):** несколько поисковых запросов в Яндексе → приоритетные сайты (Drom, Avito, Farpost и др.) → открытие страниц через Playwright → парсинг цен и объявлений → фильтр по ключевым словам и выбросам → при достаточном количестве результат сразу идёт в ответ и кеш.
- **Этап 2 (GPT):** если по Yandex мало данных — один или два запроса к GPT с веб-поиском, парсинг JSON с ценами, фильтр и удаление выбросов → min/max/avg и сохранение в кеш.
- **Итог:** в ответ клиенту/оператору попадает диапазон цен (и при необходимости разбивка по вариантам «эконом/средний/премиум»), либо эскалация, либо оценка ИИ, либо сообщение об отсутствии цен.

## Где что лежит в коде

| Этап | Файл / компонент |
|------|-------------------|
| Первое сообщение, извлечение VIN/FRAME, фото | `server/services/inbound-message-handler.ts`, `server/services/vin-ocr.service.ts` |
| Очередь и воркер поиска по VIN | `server/services/vehicle-lookup-queue.ts`, `server/workers/vehicle-lookup.worker.ts` |
| Подзамена (OEM по VIN) | `server/services/podzamenu-lookup-client.ts`, Python `podzamenu_lookup_service.py` |
| PartsAPI (декод VIN) | `server/services/partsapi-vin-decoder.ts` |
| Очередь и воркер поиска цены | `server/services/price-lookup-queue.ts`, `server/workers/price-lookup.worker.ts` |
| Определение модели КПП по OEM | `server/services/transmission-identifier.ts` |
| Поиск цен: Yandex + Playwright + GPT | `server/services/price-searcher.ts`, `server/services/price-sources/yandex-source.ts`, `server/services/playwright-fetcher.ts` |
| Кеш цен, снимки | `server/database-storage.ts` (price_snapshots, getGlobalPriceSnapshot), схема в `shared/schema.ts` |

Подробное дерево алгоритма и маппинг на узлы A0–E7 см. в **docs/TRANSMISSION_PRICE_SYSTEM_REVIEW.md** и **docs/TRANSMISSION_PRICE_IMPROVEMENT_PLAN.md**.
